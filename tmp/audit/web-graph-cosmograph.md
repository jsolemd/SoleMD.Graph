## Critical issues (security / data-loss / prod-blocker)

None confirmed. The riskiest surface is identified below as a Major concern with mitigations in place; it is not currently exploitable from untrusted input but the contract is fragile.

## Major issues (clean violations + correctness)

**Adapter boundary is documented but routinely bypassed** [/clean — adapter] — `cosmograph/index.ts:1` declares "all `@cosmograph/react` imports are contained here. Consumers import from this barrel; never from `@cosmograph/*` directly." Yet inside the slice itself the rule is broken in numerous places:
- `cosmograph/GraphRenderer.tsx:4-7` — `Cosmograph`, `CosmographRef` from `@cosmograph/react`
- `cosmograph/hooks/use-graph-selection.ts:3` — `useCosmograph`
- `cosmograph/hooks/use-points-filtered.ts:4-7` — `CosmographData`, `CosmographRef`
- `cosmograph/widgets/SelectionToolbar.tsx:11-14` — three named exports
- `cosmograph/widgets/TimelineWidget.tsx:4` — `CosmographTimeline`
- `cosmograph/widgets/use-widget-selectors.ts:4` — `useCosmograph`
- `cosmograph/widgets/FilterBarWidget.tsx:4-5` and `FilterHistogramWidget.tsx:4-5` — `FilteringClient` from a deep path `@cosmograph/cosmograph/cosmograph/crossfilter/filtering-client`, plus `Bars`/`Histogram` from `@cosmograph/ui`
- `cosmograph/widgets/init-crossfilter-client.ts:3-5` — `getInternalApi` from `@cosmograph/cosmograph/cosmograph/internal`
Fix sketch: either re-write the barrel comment to scope it to "consumers outside `cosmograph/`" (the truthful contract) or move every direct vendor import behind the barrel and keep the rule absolute. The deep `@cosmograph/cosmograph/cosmograph/internal` import is especially risky — any minor cosmograph version bump can shift those subpaths.

**Reaching into Cosmograph private internals via cast** [/clean — adapter, /audit — fragility] — `cosmograph/GraphRenderer.tsx:38-102` casts `CosmographRef` to `CosmographInternalHandle` and reaches `_cosmos.zoomInstance.behavior.transform` / `eventTransform` to read and write the d3 zoom transform. The leading underscore signals private API. Camera persistence silently degrades to `fitView` if the shape changes after a vendor bump (line 91 falsy guard), so failure is invisible. Fix sketch: file an upstream feature request for a public `getCameraTransform`/`setCameraTransform`, and in the meantime add a runtime warning on the silent fallback so a vendor break is surfaced.

**`currentPointScopeSql` is concatenated into SQL with only "trim" sanitization** [/audit — SQL injection surface] — `duckdb/sql-helpers.ts:18-24, 58-75`, used in `duckdb/queries/summary.ts:108-111`, `duckdb/queries/facets.ts:91`, `duckdb/queries/histograms.ts:69-73`, `duckdb/queries/budget.ts:90`, `duckdb/queries/node-selection.ts:127-130`, `duckdb/overlay.ts:25-26`, `duckdb/session/overlay-controller.ts` (forwarded). Today the SQL string is generated by mosaic-sql `duckDBCodeGenerator.toString(...)` from typed predicates (`lib/cosmograph-selection.ts:257-276`) so it is safe in practice. But the type system permits any caller to pass a raw `string | null` into `setCurrentPointScopeSql` (`cosmograph/hooks/use-points-filtered.ts:138`, dashboard store). There is no runtime contract that the string actually came from mosaic-sql. A future refactor that pipes user search text or URL state into this slot would silently introduce DuckDB SQL injection in a Wasm context (limited blast radius — same-origin, browser only — but still data exfil from the user's own attached `paper_documents`/`universe_points`). Fix sketch: wrap the value in a branded type (`OpaqueScopeSql`) returned only from `lib/cosmograph-selection.ts`; refuse plain strings at the store boundary and at `normalizeCurrentPointScopeSql`.

**Direct `${tableName}` interpolation in many query sites** [/audit — defense in depth] — `duckdb/queries/summary.ts:53, 110-111`, `duckdb/queries/histograms.ts:100, 271`, `duckdb/queries/budget.ts:83, 89, 242`, `duckdb/queries/facets.ts:48`, `duckdb/queries/search.ts:107`, `duckdb/queries/node-selection.ts:109, 134`. `tableName` flows from `getLayerTableName(layer)` which today returns the constant `'current_points_web'` regardless of layer (`duckdb/sql-helpers.ts:26-29`). Safe today. But `validateTableName` (`duckdb/utils.ts:48-55`) is the only fence and several call sites bypass it because they trust `getLayerTableName`. Fix sketch: route every interpolated identifier through `validateTableName` even when the source is a constant, so adding a new layer with a derived name cannot regress.

**Worker bootstrap injects URL into JS string via template literal** [/audit — code injection in worker] — `duckdb/connection.ts:110-114` builds `importScripts("${mainWorkerUrl}");` and wraps in a Blob. `mainWorkerUrl` comes from `new URL(..., window.location.origin).toString()` so it's URL-encoded — quotes survive only as `%22`. Practically safe. Risk is real if anyone refactors `resolveBrowserAssetUrl` to return a non-URL string. Fix sketch: use `JSON.stringify(mainWorkerUrl)` to guarantee escaping, or call `importScripts` from a workerized `data:` URL.

**Unbounded module-scoped widget caches** [/clean — performance, /audit — memory leak] — `cosmograph/widgets/dataset-cache.ts:8-10` declares three `Map`s (`categoricalDatasetCache`, `numericDatasetCache`, `histogramDatasetCache`) at module scope with no eviction. Keys include `bundleChecksum:layer:column:overlayRevision:scopeKey` (`getWidgetDatasetCacheKeyWithRevision:12-20`). Across overlay activations and bundle hot-swaps these grow unboundedly. `duckdb/utils.ts:6-17` already provides `createBoundedCache` — same pattern as the rest of the slice. Fix sketch: replace the raw Maps with `createBoundedCache(N)`.

**Two near-identical filter widgets duplicate ~90% of orchestration** [/clean — DRY] — `cosmograph/widgets/FilterBarWidget.tsx` and `cosmograph/widgets/FilterHistogramWidget.tsx` share: `useWidgetSelectors`, `initCrossfilterClient`, dataset-cache-with-retry pattern, scoped highlight effect, error handling, and a near-identical `useEffect` lifecycle. The second widget is 318 LOC of almost the same shape as the 307 LOC first widget. Fix sketch: extract a `useFilterWidgetLifecycle({ widgetFactory, applyDataset, applyHighlight, ... })` hook that returns `{ widget, error }` and reduces each widget file to its visual config + clause builders.

**Hard-coded prepared-statement budget heuristic without observability** [/clean — performance instrumentation] — `duckdb/queries/core.ts:14-16` (`MAX_PREPARED_STATEMENTS_PER_CONNECTION = 64`, `MAX_CACHED_STATEMENT_PLACEHOLDERS = 8`) silently churn statements when the cache is full (line 117-128). There is no telemetry to know whether real workloads hit this limit, miss the cache constantly, or whether 64 is too small. Fix sketch: emit a `__graphDebug` counter when an eviction occurs and a hit/miss ratio for prepared statements; let perf tests assert ≥ N% hit rate on the standard navigation flow.

**SelectionToolbar MutationObserver leaks if button never renders** [/audit — resource cleanup] — `cosmograph/widgets/SelectionToolbar.tsx:80-95`. The `discover` helper sets up a `MutationObserver` and returns a no-op cleanup `() => {}` instead of `() => obs.disconnect()`. The observer self-disconnects from inside the callback once an `[id]` element is found, but if the native button never renders an id child (vendor change, error state, slow load), the observer lives until React unmount with no way to stop it from outside. Fix sketch: capture `obs` in the outer scope and return `() => obs.disconnect()` from `discover`.

**`use-points-filtered.ts` swallows persistence errors silently** [/audit — error handling] — `cosmograph/hooks/use-points-filtered.ts:144`: `.catch(() => {})` on `setSelectedPointIndices`. If the DuckDB write fails (memory pressure, prepared-statement exhaustion, connection torn down mid-RAF), the UI is silently desynced from the underlying selection table — info-panel and table view will read stale `selected_point_indices`. Fix sketch: route through a centralized `reportRuntimeError` (it does not appear to exist yet — graph-level observability is missing) and at minimum increment a `__graphDebug.selectionWriteFailures` counter so regression tests can detect it.

**`bundleChecksum` is concatenated into filesystem paths server-side without validation** [/audit — defense in depth] — `lib/bundle-assets.ts:194, 223, 247-249`. `bundleChecksum` is interpolated into `path.resolve(resolvedPublishedRoot, bundleChecksum)`. Path traversal is mitigated by `assertPathWithinBundleRoot` after `realpath`, but the checksum format itself is not validated (e.g., must be `[0-9a-f]{64}`). A request with `..%2F..%2Fetc` would currently survive as a path component; `realpath` will fail and return `null`, and `recoverPublishedBundleDirectory` would attempt a DB lookup keyed by the same string (Drizzle `eq(graphRuns.bundleChecksum, …)` is parameterized). Practically safe today but brittle. Fix sketch: validate `bundleChecksum` against a strict regex at the API boundary before it ever reaches `bundle-assets.ts`.

## Minor issues

- `duckdb/persistent-cache.ts:77-91` — `UPDATE … WHERE cache_slot = 'hot_bundle'` interpolates the constant via template literal even though `executeStatement(?, …)` is right there. Use parameters consistently.
- `cosmograph/GraphRenderer.tsx:417` — `container.querySelector<SVGSVGElement>("svg")` then arbitrarily mutates `style.zIndex`. If Cosmograph internally re-renders the brush SVG, the mutation is lost. The effect runs once on mount only. Consider a `MutationObserver` or a CSS rule keyed by an attribute.
- `cosmograph/GraphRenderer.tsx:498-499` — `hoveredPointLabelClassName=""` and `pointLabelFontSize={11}` are magic literals; both belong with the rest of the label theme tokens.
- `duckdb/queries/core.ts:198-228` — `buildReadOnlyQuery` allows multiple read keywords but rejects only `;`. A `WITH x AS (SELECT 1) PRAGMA enable_object_cache` would parse but fail; the whitelist trusts the first token only. This is fine because the wrapper still rejects writes via DuckDB itself, but the docstring should call out that statement-shape validation is delegated to DuckDB.
- `lib/colors.ts:188-191` — `viridis` and `plasma` palettes duplicate their first 10 entries to fill 20 slots; if those legacy schemes are truly retired, drop the rows; if not, document why duplication is the intended UX.
- `cosmograph/widgets/FilterBarWidget.tsx:170, FilterHistogramWidget.tsx:183` — fixed retry delay loop `[0, 150, 450]` for empty datasets is a polling smell; ideally the producer signals "ready" rather than retrying blindly.
- `lib/selection-query-state.ts:5-14` — `normalizeCurrentPointScopeSql` only trims whitespace; the name suggests deeper normalization. Either rename to `trim…` or actually validate predicate well-formedness here (would also harden the major issue above).
- `duckdb/queries/histograms.ts:266` — `'${escapeSqlLiteral(column)}' AS column_key` works because column names are validated upstream, but this is the "belt and suspenders" pattern; document why both layers are present so a future cleanup does not drop one.

## Reuse / consolidation opportunities

- **Filter widgets**: `FilterBarWidget` + `FilterHistogramWidget` should share a `useFilterWidgetLifecycle` hook (see Major issue). Today they share `useWidgetSelectors`, `initCrossfilterClient`, and `dataset-cache` but each maintains its own copy of dataset/highlight/error/revision state.
- **Selection-source guards**: `cosmograph/lib/cosmograph-selection.ts` defines `isVisibilitySelectionSourceId`, `isBudgetScopeSelectionSourceId`, `matchesSelectionSourceId`, etc. These are re-implemented inline in `use-points-filtered.ts` (line 51-62 inlines `getIntentClauseIds`). Extract `getIntentClauseIds(selection)` into the lib module.
- **Adapter unification**: every cosmograph handle import (`useCosmograph`, `CosmographRef`, `CosmographData`, `CosmographTimeline`, `CosmographButton*`) should re-export from `cosmograph/index.ts`. Right now the barrel re-exports a curated set from `@solemd/graph/cosmograph` but never the raw `@cosmograph/react` symbols that intra-slice files actually use.
- **SQL identifier validation**: `validateTableName`, `resolveInfoColumn`, `resolveSearchColumn`, and `getLayerTableName` are scattered across `utils.ts` and `sql-helpers.ts`. Consolidate into a single `sql-identifiers.ts` so future query files have one obvious place to import from.
- **`escapeSqlString` vs `escapeSqlLiteral`**: same implementation in `duckdb/queries/core.ts:33-35` and `duckdb/sql-helpers.ts:77-79`. Pick one.

## What's solid

- **Parameterized prepared-statement infrastructure** (`duckdb/queries/core.ts`): explicit serialization queue per connection (`enqueueConnectionRead`), bounded prepared-statement cache with eviction-on-error, careful close handling. Best primitive in the slice.
- **`createBoundedCache`/`cachedQuery`** (`duckdb/utils.ts`): clean cache-aside pattern with eviction-on-error and optional `evictWhen`. Used pervasively in `query-controller.ts`. Worth replicating in `widgets/dataset-cache.ts`.
- **Selection clause builders** (`lib/cosmograph-selection.ts`): everything goes through mosaic-sql typed predicates and `duckDBCodeGenerator.toString`. This is the right architectural boundary — the only thing missing is a runtime brand to enforce it.
- **`registerInitialSessionViews`** (`duckdb/views/register-all.ts`): named bootstrap steps wrapped in `runBootstrapStep` for consistent error context. Good pattern.
- **`closeConnection`** (`duckdb/connection.ts:150-182`): collects the first error across multiple cleanup phases instead of swallowing or short-circuiting. Worth replicating elsewhere.
- **Test coverage**: 51 test files for ~86 source files. Render stability, widget baselines, persistent cache, attachment, query controller, info queries are all covered. Few slices in this codebase are this thoroughly tested.
- **Visibility-change RAF retry** (`cosmograph/GraphRenderer.tsx:394-409`): correctly handles the "tab backgrounded during load" RAF-pause edge case. Documented.
- **Stable `onPointsFiltered` handler** (`cosmograph/hooks/use-points-filtered.ts:217-223`): ref-based wrapper avoids re-registering Cosmograph callbacks on dependency changes — a real perf win on a 15-dep handler.

## Recommended priority (top 5 ordered)

1. **Make adapter contract real**: either re-scope the `cosmograph/index.ts` claim to "consumers outside this folder" or actually route every `@cosmograph/*` import through the barrel. Top priority because the deep `@cosmograph/cosmograph/cosmograph/internal` import in `init-crossfilter-client.ts` is a vendor-version bomb and the boundary is a load-bearing /clean rule.
2. **Brand `currentPointScopeSql`**: introduce an opaque type minted only by `lib/cosmograph-selection.ts`; refuse raw strings at `setCurrentPointScopeSql` and `normalizeCurrentPointScopeSql`. Closes the SQL-injection failure mode without disrupting any current call site.
3. **Bound the widget dataset caches** (`cosmograph/widgets/dataset-cache.ts`): replace the raw `Map`s with `createBoundedCache`. One-line fix that closes a real memory leak across long sessions and bundle hot-swaps.
4. **Extract a shared filter-widget lifecycle hook**: collapse `FilterBarWidget` + `FilterHistogramWidget` orchestration. Cuts ~250 LOC of duplication and removes drift risk between the two widgets' selection/highlight semantics.
5. **Add observability for silent failure paths**: at minimum surface `selectionWriteFailures`, `preparedStatementEvictions`, and the `applyViewportCamera` private-API fallback through `__graphDebug`. Without these, regressions in the cosmograph-internal handle, prepared-statement budget, or selection persistence are invisible until a user reports stale UI.
