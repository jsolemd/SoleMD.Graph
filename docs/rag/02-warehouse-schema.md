## Purpose

Define the fastest, most scalable PG 18 warehouse schema possible for a
14 M-paper biomedical knowledge graph with clean-slate authoring, so that
every downstream surface (ingest, projection, serving) resolves its types,
keys, and joins against the same contract.

## 0. Global conventions

Everything under this heading applies to every table in the warehouse
unless a table explicitly states otherwise.

### 0.1 Types

| Concern | Choice | Rationale |
|---|---|---|
| Time | `timestamptz` universally | UTC-anchored, locale-safe, no `timestamp` ambiguity. |
| Variable text | `text` only; never `varchar(n)` | `varchar(n)` has no PG performance benefit and introduces schema churn on length changes. |
| Semi-structured | `jsonb` only; never `json` | Binary storage, indexable, fast access. |
| Dates without time | `date` | Source-native precision when the upstream only gives a date. |
| Money / exact decimal | `numeric` | None today; reserved. |
| Scores / weights | `real` (single precision) | Ranking signals don't need double precision; halves storage. |
| Dense embeddings | `halfvec(n)` (pgvector) | 50 % storage vs `vector(n)`, recall parity at 768 dims, 29× faster parallel HNSW build. Provisional pending sample. |
| Enums | `smallint` with constant mapping in the engine code, **not** PG `ENUM` | PG `ENUM` adds DDL on every new value; `smallint` + named constants is cheaper and more portable. |
| Bit flags | `bit(n)` | Compact source-mask style bitfields (`source_mask`). |

### 0.2 Identity

| Identity | Type | Generator | Lives on |
|---|---|---|---|
| `corpus_id` | `bigint` | `IDENTITY GENERATED BY DEFAULT AS IDENTITY` | every fact table keyed to a paper |
| `concept_id` | `bigint` | identity | `concepts`, fact tables |
| `venue_id`, `author_id` | `bigint` | identity | registries |
| `source_release_id` | `integer` | identity (low cardinality) | `source_releases`, fact tables |
| `ingest_run_id` | `uuid` (v7) | `uuidv7()` (PG 18 native) | `ingest_runs`, `load_history` |
| `graph_run_id` | `uuid` (v7) | `uuidv7()` | `graph_runs` |
| `chunk_version_key` | `uuid` (v7) | `uuidv7()` | `paper_chunk_versions` |
| `api_projection_run_id`, `serving_run_id` | `uuid` (v7) | `uuidv7()` | serve cluster (in `03`) |
| `evidence_key` | `uuid` (v5) | deterministic `uuidv5(solemd_ns, …)` | `paper_evidence_units` |

Rationale:
- `BIGINT` everywhere rows are counted in the tens of millions or more and
  frequently join — tight (8 B), sequential, cache-friendly.
- `UUIDv7` on run / version identities that cross system boundaries
  (projection jobs, serving packages, graph builds) so they're
  timestamp-ordered and externally sortable without leaking internal
  sequence values.
- `UUIDv5` on `evidence_key` so grounding identity is content-bound,
  release-independent, and round-trippable into PG — the whole point per
  `rag-future.md` Identity Glossary.

### 0.3 Column layout (MAXALIGN ordering)

Every table orders columns in fixed-width-descending → variable-last:

```
8-byte (BIGINT, DOUBLE PRECISION, TIMESTAMPTZ, UUID)
  → 4-byte (INT, REAL, DATE)
    → 2-byte (SMALLINT)
      → 1-byte (BOOL, CHAR, "char")
        → variable (TEXT, JSONB, arrays, vectors, tsvector, bit varying)
```

At 100 M-row scale this saves 8–16 bytes per tuple (0.8–1.6 GB per 100 M
rows) on alignment padding. Worth doing once, on a fresh build.

### 0.4 TOAST & compression

- Cluster-wide: `default_toast_compression = lz4`. LZ4 is 2–4× faster
  decompression than pglz with similar ratios for English prose.
- `text` columns inherit `EXTENDED` (compress + TOAST out-of-line).
- Columns that are *always* read in full (for example, `paper_text.abstract`
  when served to the frontend) set `STORAGE EXTERNAL` (TOAST out-of-line,
  no compression) to save a decompression round-trip per read.
- `toast_tuple_target` stays at the PG 18 default (2 kB) except where a
  single wide table has one rarely-read large column we want to keep off
  the main heap page.

### 0.5 Fillfactor tiering

| Tier | Value | Applies to |
|---|---:|---|
| Append-mostly large facts | 100 | `paper_citations`, `paper_citation_contexts`, `paper_concepts`, `paper_relations`, `paper_blocks*`, `paper_sentences*`, `paper_*_mentions*`, `paper_chunk_members*`, `pubtator.*` |
| Update-occasional | 90 | `papers`, `paper_text`, `paper_lifecycle`, `paper_metrics`, `concepts`, `concept_aliases`, `concept_search_aliases` |
| Status-flipping control | 80 | `source_releases`, `ingest_runs`, `paper_chunk_versions`, `graph_runs`, `graph_bundle_artifacts` |

Rationale: HOT updates need free space on the same page; append-mostly
tables don't benefit from fillfactor < 100, and going below 80 is
empirically worse than 80 in modern PG.

### 0.6 Partitioning

- **Hash partition count**: 32 for every `corpus_id`-keyed fact family.
  32 × (PK + 1–2 local indexes) stays within PG 18's 64-slot fast-path
  lock table; 64 partitions × 3 indexes lands in the main lock table and
  produces `LWLock:LockManager` contention on this box's 24 logical
  cores. Provisional pending sample.
- **No partitioning** for small / bounded tables (`corpus`, `papers`,
  `paper_text`, `venues`, `authors`, `concepts`, `graph_runs`, etc.) —
  they never grow past a few tens of millions of rows, fit comfortably in
  a single table, and partitioning would only add planner cost.
- **Time-range partitioning** (`range` on a `timestamptz` or
  `source_release_id`): considered for `ingest_runs` and `load_history`
  if they grow past ~10 M rows. Today they don't; revisit via
  `pg_partman` 5.x when measured.

### 0.7 Indexing principles

- PK on every table. No heap-only tables.
- One canonical forward index per join direction. One canonical reverse
  index when both directions are hot.
- Partial indexes for hot filter predicates (`is_influential = true`,
  `retraction_status = 'active'`, `is_active = true`, `is_primary = true`).
- GIN for `tsvector`; GiST + `gin_trgm_ops` for trigram fuzzy lookup on
  small canonical alias tables.
- BRIN on append-time-ordered columns with physical correlation
  (`source_releases.source_ingested_at`, `ingest_runs.started_at`) —
  cheap and effective on analytical scans.
- `INCLUDE` covering indexes only on serve-side list queries
  (they live in `03-serve-schema.md`, not here).
- No speculative ANN indexes in PG (per `rag-future.md` §1516 — the
  serving plane is OpenSearch). Warehouse `paper_embeddings_retrieval`
  gets no HNSW by default.

### 0.8 Schemas

Three schemas only:

- `solemd` — canonical warehouse, grounding, concepts, build control.
- `pubtator` — raw PubTator3 ingest surfaces.
- `umls` — raw UMLS Metathesaurus reference tables.

No `raw`, `stage`, or `archive` schemas — the `s2_*_raw` tables already
carry the raw-ingest role, and stage-and-swap uses `_next` / `_old`
suffixes on the same schema.

### 0.9 Tablespaces

**Do not create tablespaces inside either cluster.** Per `00-topology.md`,
storage isolation is the responsibility of the two-cluster split, not of
tablespaces within a cluster.

### 0.10 Comments and migrations

- `db/schema/enum-codes.yaml` is the authoritative machine-readable
  registry for every SMALLINT-coded enum used by warehouse and serve.
  `engine/app/models/shared/enums.py` and the generated SQL
  column-comment artifact are **exclusively generated from it**; docs
  use symbolic names for readability, but on-disk values are SMALLINT
  codes from the registry.
- Every table and every non-obvious column carries a `COMMENT ON …` so
  `\d+` output and tooling stay legible.
- Ordered PostgreSQL SQL under `db/schema/warehouse/` is the authoring surface;
  `scripts/schema_migrations.py` remains the executor ledger
  per `12-migrations.md`.

## 1. Extensions

Required at warehouse cluster boot:

```sql
CREATE EXTENSION IF NOT EXISTS vector;            -- halfvec, vector, ivfflat, hnsw
CREATE EXTENSION IF NOT EXISTS pg_trgm;           -- trigram GiST / GIN
CREATE EXTENSION IF NOT EXISTS pgcrypto;          -- digest, hmac helpers
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;-- baseline observability
CREATE EXTENSION IF NOT EXISTS pg_buffercache;    -- warmup + hit-ratio diag
CREATE EXTENSION IF NOT EXISTS hypopg;            -- plan experiments
CREATE EXTENSION IF NOT EXISTS pg_cron;           -- scheduled VACUUM, refresh
CREATE EXTENSION IF NOT EXISTS pg_partman;        -- only if time-range lifecycle kicks in
```

`postgres_fdw` is **not** installed on the warehouse cluster. FDW lives
on the serve cluster reading *into* warehouse; the warehouse cluster is
always the source, never the querier.

`auto_explain` is a preload module, not a `CREATE EXTENSION` surface.
Load it via `shared_preload_libraries` / `session_preload_libraries` and tune
`auto_explain.*` in `09-tuning.md`. Day-one UUIDv5 generation for
`evidence_key` is engine-owned; if DB-side UUIDv5 audit helpers are later
needed, add `uuid-ossp` explicitly rather than assuming a core `uuidv5()`
function.

## 2. Identity glossary (warehouse-scoped)

| Identity | Meaning | Lifetime | Generation |
|---|---|---|---|
| `corpus_id` | Canonical paper identity across the entire SoleMD estate. | Permanent. Never recycled. | Identity sequence on `solemd.corpus.corpus_id`. Every other warehouse table that references a paper carries `corpus_id`, not `pmid` / `doi` / `s2_paper_id`. |
| `source_release_id` | Surrogate PK for one source release (e.g. S2 `2026-03-10`). | Permanent. | Identity sequence; low cardinality. |
| `ingest_run_id` | One ingest run against one source release. | Permanent. | `uuidv7()` — timestamp-ordered so `ORDER BY run_id` tracks wall-clock order. |
| `concept_id` | Canonical concept identity across RAG, graph, and API surfaces. | Permanent. | Identity sequence on `solemd.concepts`. Source identifiers (UMLS CUI, MeSH, NCBI Gene) live in `concept_xrefs`. |
| `chunk_version_key` | Version identity for the chunk derivation policy. | Permanent. | `uuidv7()` at policy-change time. |
| `graph_run_id` | Graph build run identity. | Permanent. | `uuidv7()`. |
| `evidence_key` | Content-bound grounding identity. Round-trippable to canonical coordinates via `paper_evidence_units`. | Permanent for a given `chunk_version_key`. | Deterministic `uuidv5(solemd_namespace, corpus_id ‖ kind ‖ section_ord ‖ block_ord ‖ sent_start ‖ sent_end ‖ chunk_version_key)`. |

Derived identities (`api_projection_run_id`, `serving_run_id`) live on the
serve cluster — see `03-serve-schema.md`.

## 3. Partitioning contract

### 3.1 Families

Hash-partitioned by `corpus_id` into 32 children, named `_p00 … _p31`:

- `solemd.paper_blocks`
- `solemd.paper_sentences`
- `solemd.paper_citation_mentions`
- `solemd.paper_entity_mentions`
- `solemd.paper_chunk_members`
- `solemd.paper_chunks` (compat / fallback)
- `solemd.paper_concepts`
- `solemd.paper_relations`
- `pubtator.entity_annotations` (after `pmid → corpus_id` backfill)
- `pubtator.relations` (after backfill)

Hash-partitioned by `citing_corpus_id` into 32 children:

- `solemd.paper_citations`
- `solemd.paper_citation_contexts`

### 3.2 Children

Every child declares:
- Local PK matching the parent.
- `fillfactor = 100` (all partitioned families are append-mostly).
- Local indexes identical in structure across children.
- `autovacuum_enabled = false` **during bulk load only**, re-enabled after.

The canonical SQL schema files under `db/schema/warehouse/` enumerate all
32 children explicitly. This document shows parent + one representative child;
read the schema files for the full set.

### 3.3 Staged partitioning for `pubtator.*`

Until the `pmid → corpus_id` backfill completes, raw PubTator rows lack
a `corpus_id`. Partitioning plan:

1. Initial load into a non-partitioned staging table
   `pubtator.entity_annotations_stage`.
2. Backfill `corpus_id` via join to `solemd.papers(pmid)`.
3. COPY backfilled rows into the partitioned parent
   `pubtator.entity_annotations`.
4. Drop staging after verification.

This avoids a "half the rows don't prune" regression that
list-then-hash nested partitioning would cause.

## 4. Table families

Each subsection gives a one-paragraph purpose, the canonical column
layout (in MAXALIGN order, with types), the keys / indexes, partitioning
(if any), fillfactor, and representative SQL for the parent. Children and
secondary indexes are noted in prose where omitting them keeps the doc

### 4.1 Raw ingest and release tracking

Raw-ingest tables exist to preserve source-normalized inputs in PG
without making them the runtime surface. They are canonical *rebuild
inputs*, not canonical truth.

#### `solemd.source_releases`

One row per external-source release (S2 monthly drops, PubTator3 weekly,
UMLS quarterly). Fillfactor 80 (status transitions).

```hcl
table "source_releases" {
  schema = schema.solemd
  column "source_release_id"    { null = false, type = integer, identity { generated = BY_DEFAULT } }
  column "source_published_at"  { null = true,  type = timestamptz }
  column "source_ingested_at"   { null = false, type = timestamptz, default = sql("now()") }
  column "manifest_checksum"    { null = true,  type = text }
  column "manifest_uri"         { null = true,  type = text }
  column "source_name"          { null = false, type = text }
  column "source_release_key"   { null = false, type = text }
  column "release_status"       { null = false, type = text, default = "planned" }  // planned | ingesting | loaded | retired
  primary_key { columns = [column.source_release_id] }
  index "uq_source_releases_natural" {
    unique  = true
    columns = [column.source_name, column.source_release_key]
  }
  index "idx_source_releases_ingested" {
    columns = [column.source_name, column.source_ingested_at]
    // used for "latest release per source" lookups
  }
  settings { fillfactor = 80 }
}
```

#### `solemd.ingest_runs`

One row per ingest or rebuild run. Fillfactor 80. Symbolic lifecycle and
operator-action names below are stored as SMALLINT codes from the enum
registry in §0.10.

Columns (MAXALIGN order):
- `ingest_run_id` UUID, PK, default `uuidv7()`
- `advisory_lock_key` BIGINT
- `started_at`, `completed_at` TIMESTAMPTZ
- `source_release_id` INTEGER, FK → `source_releases`
- `status` SMALLINT (`started` | `loading` | `indexing` | `analyzing` |
  `published` | `failed` | `aborted`)
- `requested_status` SMALLINT (`continue` | `abort`) nullable
- `manifest_uri` TEXT
- `error_message` TEXT
- `last_loaded_family` TEXT
- `families_loaded` TEXT[] default empty array
- `plan_manifest` JSONB
- `phase_started_at` JSONB

Indexes:
- PK on `ingest_run_id`.
- Btree on `(source_release_id, started_at DESC)` — "latest run per
  release."
- Btree on `(status, started_at DESC)` — stuck-run and recent-failure
  operational lookups.
- BRIN on `started_at` — analytical scans.

#### `solemd.s2_papers_raw`

Normalized S2 paper metadata. Fillfactor 100 (append-only per release).

Columns: `corpus_id` (nullable until mapped), `source_release_id`,
`paper_id` TEXT (S2 paperId), `pmid` INTEGER, `doi_norm` TEXT, `pmc_id`
TEXT, `title` TEXT, `abstract` TEXT, `venue_raw` TEXT, `year` INTEGER,
`publication_date` DATE, `payload_checksum` TEXT, `last_seen_run_id` UUID.

Keys / indexes:
- PK `(paper_id)`.
- Btree `(pmid)` where `pmid IS NOT NULL`.
- Btree `(doi_norm)` where `doi_norm IS NOT NULL`.
- Btree `(pmc_id)` where `pmc_id IS NOT NULL`.
- Btree `(corpus_id)` where `corpus_id IS NOT NULL` — used during backfill.

Rule: raw fields are typed and narrow. We do **not** store giant JSONB
payload blobs as the primary raw representation; the original S2 file
is preserved under `/mnt/solemd-graph/data/semantic-scholar/releases/…`
and referenced by `manifest_uri` on `ingest_runs`.

#### `solemd.s2_paper_authors_raw`

- `paper_id` TEXT, `author_ordinal` INTEGER, `source_author_id` TEXT,
  `name_raw` TEXT, `affiliation_raw` TEXT.
- PK `(paper_id, author_ordinal)`.
- Btree `(source_author_id)`.

#### `solemd.s2_paper_references_raw`

- `citing_paper_id` TEXT, `cited_paper_id` TEXT, `first_seen_release_id`
  INTEGER, `linkage_status` SMALLINT (`pending` | `linked` | `orphan`),
  `is_influential` BOOLEAN, `intent_raw` TEXT.
- PK `(citing_paper_id, cited_paper_id)`.
- Btree `(linkage_status, citing_paper_id)`.
- Reverse btree `(cited_paper_id, citing_paper_id)`.

#### `solemd.s2_paper_assets_raw`

- `paper_id` TEXT, `asset_kind` TEXT, `asset_url` TEXT,
  `content_type` TEXT, `availability_raw` TEXT, `asset_checksum` TEXT.
- PK `(paper_id, asset_kind, asset_url)`.
- Btree `(paper_id)`.

#### `pubtator.entity_annotations`

Hash-partitioned by `corpus_id` × 32 after the `pmid → corpus_id`
backfill. Fillfactor 100. Append-mostly per release.

Columns (MAXALIGN order):
- `corpus_id` BIGINT NOT NULL (post-backfill)
- `source_release_id` INTEGER NOT NULL
- `start_offset` INTEGER NOT NULL
- `end_offset` INTEGER NOT NULL
- `pmid` INTEGER (retained for audit and for un-backfilled rows)
- `entity_type` SMALLINT NOT NULL (mapped from PubTator type)
- `mention_text` TEXT NOT NULL
- `concept_id_raw` TEXT NOT NULL (source CUI / MeSH / NCBI Gene)
- `resource` SMALLINT NOT NULL (source mask: PubTator3, etc.)

Local indexes per partition:
- PK `(corpus_id, start_offset, end_offset, concept_id_raw)`.
- Btree `(corpus_id, concept_id_raw)` — concept-first scans.
- Btree `(pmid, start_offset)` — used until backfill is complete; can
  drop after.

#### `pubtator.relations`

Hash-partitioned by `corpus_id` × 32. Fillfactor 100.

Columns: `corpus_id`, `source_release_id`, `pmid`, `relation_type` SMALLINT,
`subject_entity_id` TEXT, `object_entity_id` TEXT, `subject_type` SMALLINT,
`object_type` SMALLINT, `relation_source` SMALLINT.

Local indexes per partition:
- PK `(corpus_id, subject_entity_id, relation_type, object_entity_id)`.
- Btree `(corpus_id, object_entity_id, relation_type, subject_entity_id)`.

#### `umls.*`

UMLS Metathesaurus raw tables — `MRCONSO`, `MRREL`, `MRSTY`, `MRHIER`,
`MRDEF`. These keep the UMLS source-native schema (not MAXALIGN-reordered)
so a re-load from Metathesaurus drops in cleanly. All are
`fillfactor = 100`. SoleMD-canonical concept derivation lives in
`solemd.concepts` and `solemd.concept_*`, not here.

### 4.2 Canonical metadata & lifecycle

The metadata layer is narrow, typed, and joinable. It is **not** where
text or vectors live — those split out into `paper_text` and
`paper_embeddings_*`.

#### `solemd.corpus`

Stable domain membership. Fillfactor 90.

```hcl
table "corpus" {
  schema = schema.solemd
  column "corpus_id"        { null = false, type = bigint, identity { generated = BY_DEFAULT } }
  column "first_seen_at"    { null = false, type = timestamptz, default = sql("now()") }
  column "last_seen_at"     { null = false, type = timestamptz, default = sql("now()") }
  column "admission_reason" { null = false, type = text }
  column "domain_status"    { null = false, type = text, default = "candidate" }  // candidate | mapped | retired
  primary_key { columns = [column.corpus_id] }
  index "idx_corpus_domain_status" {
    columns = [column.domain_status]
  }
  settings { fillfactor = 90 }
}
```

Rule: release-scoped flags (`is_in_current_map`, `is_in_current_base`)
that lived in `corpus` previously are removed. Current-run membership
lives on `graph_points` / `serving_members` in the serve cluster.

#### `solemd.papers`

Canonical paper identity + non-text bibliographic metadata. Fillfactor 90.

Columns (MAXALIGN):
- `corpus_id` BIGINT PK FK → `corpus`
- `venue_id` BIGINT FK → `venues`
- `publication_date` DATE
- `year` SMALLINT (denormalized for range scans without casting)
- `language` SMALLINT (enum code)
- `article_type` SMALLINT (enum code)
- `is_open_access` BOOLEAN
- `is_retracted` BOOLEAN NOT NULL DEFAULT false
- `pmid` INTEGER unique-where-not-null
- `doi_norm` TEXT unique-where-not-null
- `pmc_id` TEXT unique-where-not-null
- `s2_paper_id` TEXT unique-where-not-null
- `created_at`, `updated_at` TIMESTAMPTZ

Indexes:
- PK `(corpus_id)`.
- Unique partial btrees on each external id (pmid, doi, pmc, s2).
- Btree `(venue_id, year)` only if query families actually need it
  (provisional — drop if not used).

#### `solemd.paper_text`

Canonical text storage, split out from `papers` so metadata reads don't
drag text payloads off disk. Fillfactor 90.

Columns (MAXALIGN):
- `corpus_id` BIGINT PK FK → `corpus`
- `title_hash` BYTEA (16 B SHA-1 prefix of normalized title — exact-match fast path)
- `abstract_hash` BYTEA
- `text_availability` SMALLINT (enum: none | abstract | fulltext)
- `title` TEXT NOT NULL
- `normalized_title_key` TEXT NOT NULL
  `GENERATED ALWAYS AS (solemd.normalize_title_key(title)) STORED`
- `abstract` TEXT (STORAGE EXTERNAL — always-full-read)
- `tldr` TEXT
- `fts_vector` TSVECTOR `GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(abstract, '')), 'B')
  ) STORED`

Indexes:
- PK `(corpus_id)`.
- Btree `(normalized_title_key)` — exact title matching.
- GiST `(title gist_trgm_ops)` — fuzzy title lookup (deferred until
  proven needed).
- GIN `(fts_vector)` with `fastupdate = on`, `gin_pending_list_limit =
  16 MB`. Post-bulk-load runs `gin_clean_pending_list()` before
  publishing.
- Btree `(text_availability, corpus_id)` — partial for
  `text_availability > 0` to find text-bearing papers.

#### `solemd.venues`

- `venue_id` BIGINT identity PK.
- `issn`, `eissn` TEXT.
- `display_name`, `normalized_name` TEXT.
- `curated_family` SMALLINT (enum) — for graph / retrieval priors.
- Fillfactor 90. Unique btree on `normalized_name`.

#### `solemd.authors`

- `author_id` BIGINT identity PK.
- `orcid` TEXT unique-where-not-null.
- `display_name`, `normalized_name` TEXT.
- Fillfactor 90. Btree on `normalized_name`.

#### `solemd.paper_authors`

- `corpus_id` BIGINT FK → `corpus`.
- `author_id` BIGINT FK → `authors`.
- `author_ordinal` SMALLINT NOT NULL.
- `affiliation_text` TEXT (nullable, EXTERNAL).
- PK `(corpus_id, author_ordinal)`. Reverse btree `(author_id, corpus_id)`.
- Fillfactor 100 (append-only once publication is canonical).

#### `solemd.paper_assets`

- `corpus_id`, `asset_kind` SMALLINT, `asset_uri` TEXT, `source_name` SMALLINT,
  `availability_state` SMALLINT, `rights_class` SMALLINT,
  `license_kind` SMALLINT, `excerpt_eligible` BOOLEAN,
  `fulltext_eligible` BOOLEAN, `asset_checksum` TEXT.
- PK `(corpus_id, asset_kind, asset_uri)`. Btree `(corpus_id)`.
- Fillfactor 100.

#### `solemd.paper_lifecycle`

Retraction / correction / serving-eligibility governance. Fillfactor 90.

Columns: `corpus_id` PK, `first_seen_source_release_id` INTEGER,
`last_seen_source_release_id` INTEGER, `last_refreshed_at` TIMESTAMPTZ,
`retraction_status` SMALLINT, `retracted_at` TIMESTAMPTZ,
`correction_status` SMALLINT, `correction_source` TEXT,
`tombstone_status` SMALLINT, `tombstone_reason` SMALLINT,
`serving_rights_class` SMALLINT, `serving_eligibility_class` SMALLINT.

Indexes:
- PK.
- Partial btree `(serving_eligibility_class, corpus_id)` where
  `serving_eligibility_class > 0` — hot-lane admission queries.
- Btree `(retraction_status, corpus_id)` where `retraction_status > 0`
  — retraction audit.
- Btree `(last_seen_source_release_id, corpus_id)` — stale-paper scans.

### 4.3 Canonical concepts & references

### 4.6 Canonical graph embeddings

Graph-build embeddings live in their own table so `papers` / `paper_text`
stays narrow. **Retrieval embeddings are not stored in warehouse PG** —
OpenSearch is the runtime owner per `rag-future.md`, and reproducibility
comes from release-scoped archive artifacts on `/mnt/solemd-graph`
(see `01-storage.md` §4), not from a relational table.

#### `solemd.paper_embeddings_graph`

- `corpus_id` BIGINT
- `model_key` SMALLINT (e.g. `SPECTER2_v2 = 1`)
- `embedding_revision` SMALLINT
- `embedding_source_kind` SMALLINT (`upstream_release = 1`, `local_generated = 2`)
- `embedding` halfvec(768)

- PK `(corpus_id, model_key)`. No ANN index by default — SPECTER2 powers
  graph build (cuML / cuGraph) and relatedness priors, not runtime
  retrieval.

Rule: this is a warehouse-side graph-build / audit table only. ANN
indexing stays on OpenSearch for runtime dense retrieval; graph build
runs its own ANN outside PG via cuML / cuGraph. If a PG-local
comparison path is ever justified, HNSW on this table is the trigger
already listed in the deferred decisions.

Provenance rule: graph embeddings must be distinguishable as upstream-provided
versus locally generated. `model_key` + `embedding_revision` name the model
family/version; `embedding_source_kind` records whether the row came from an
upstream release shard or from local generation during a rollout wave.

## 5. Cross-family invariants

Enforced by a mix of foreign keys, generated columns, and scheduled
audit queries in `pg_cron`.

1. `corpus_id` present on every fact, grounding, concept-attribution,
   and embedding row joins back to exactly one `solemd.corpus` row.
2. `evidence_key` in `paper_evidence_units` is deterministic from
   `(corpus_id, evidence_kind, section_ordinal, block_ordinal,
   sentence_start_ordinal, sentence_end_ordinal, chunk_version_key)`.
   A scheduled audit recomputes the hash and asserts equality.
3. `concept_id_raw` on PubTator / mention rows maps to exactly one
   `concept_id` in `concepts` via `concept_xrefs` once the backfill is
   complete. Unmapped rows carry `concept_id IS NULL` (not a sentinel
   value).
4. `source_release_id` values on facts are all `<=` the
   `last_seen_release_id` on `paper_lifecycle.corpus_id`.
5. `paper_text.fts_vector` is always recomputed from current
   `(title, abstract)` — enforced by `GENERATED ALWAYS AS … STORED`,
   not a trigger.

## 6. Write patterns

### 6.1 Bulk ingest window

Per `05-ingest-pipeline.md`:
1. Target table set to `UNLOGGED` for the run.
2. Autovacuum disabled on target partitions.
3. Indexes dropped (or created on empty table then data loaded — PG 18
   permits both; dropping is cheaper at TB scale).
4. DuckDB transforms parquet → binary COPY, fans out to 32
   per-partition asyncpg `copy_records_to_table` streams.
5. `CREATE INDEX` (parallel, not `CONCURRENTLY`) using
   `maintenance_work_mem=16GB`, `max_parallel_maintenance_workers=12`.
6. `ALTER TABLE … SET LOGGED` per partition.
7. `VACUUM (FREEZE, ANALYZE, PARALLEL 6)`.
8. Autovacuum re-enabled with per-table thresholds below.

### 6.2 Incremental write

- Canonical metadata (`papers`, `paper_text`, `paper_lifecycle`,
  `concepts`): `INSERT … ON CONFLICT … DO UPDATE` keyed on the natural
  identity.
- Fact tables (`paper_citations`, `paper_concepts`): incremental loads
  stage into `_next` partition-matching tables, then `ATTACH PARTITION`
  replaces the stale month. No in-place UPDATE of fact rows.
- Control tables (`source_releases`, `ingest_runs`, `graph_runs`):
  small UPDATE traffic; autovacuum default.

### 6.3 Per-table autovacuum overrides

Append-mostly large facts:
```sql
ALTER TABLE solemd.paper_citations SET (
  autovacuum_vacuum_scale_factor = 0.2,
  autovacuum_vacuum_insert_scale_factor = 0.5,
  autovacuum_analyze_scale_factor = 0.1
);
```

Text-bearing grounding partitions: same as above plus
`autovacuum_vacuum_cost_delay = 2` so vacuum doesn't starve ingest.

Control tables:
```sql
ALTER TABLE solemd.source_releases SET (
  autovacuum_vacuum_scale_factor = 0.0,
  autovacuum_vacuum_threshold = 50
);
```

## 7. Read patterns

### 7.1 From the serve cluster (engine API)

- Bounded grounding dereference over `postgres_fdw`. Permitted targets
  and key shapes are in `00-topology.md` §4. Never a full-table scan
  across FDW.

### 7.2 From warehouse-local projection jobs

- Large hash-joins over `paper_citations` / `paper_entity_mentions` /
  `pubtator.*` happen here, not on serve.
- Every projection job reads warehouse + writes serve (staging table +
  swap). Projection jobs never write warehouse beyond their own control
  rows in `api_projection_runs` (on serve).

### 7.3 Analytical queries

- Ad-hoc analytical runs (benchmark evaluation, concept drift audit)
  execute on warehouse directly through the admin pool. Never through
  the serving path.

## 8. Decisions — locked / provisional / deferred

### Locked now

| Decision | Rationale |
|---|---|
| Three schemas only: `solemd`, `pubtator`, `umls` | Fewer schemas = fewer search-path issues; raw-ingest tables live under `solemd.s2_*_raw` without a separate `raw` schema. |
| BIGINT `corpus_id` with identity sequence | Single-writer single-DB; tight join keys; cannot be undone after ingest. |
| UUIDv7 for `ingest_run_id`, `graph_run_id`, `chunk_version_key` | Timestamp-ordered, externally sortable, no sequence leakage. |
| UUIDv5 for `evidence_key`, stored in `paper_evidence_units` | Content-bound, release-independent, round-trippable from serving docs. |
| Hash partitioning by `corpus_id` / `citing_corpus_id` × 32 for the grounding + citation + entity families | Matches rag-future.md §4 shape; stays inside PG 18 fast-path lock slots. |
| `paper_text` split out from `papers` | Metadata reads don't drag abstracts off disk. |
| `paper_citation_contexts` split out from `paper_citations` | Narrow edge, heavy text separated. |
| `paper_embeddings_graph` kept; `paper_embeddings_retrieval` omitted from day-one warehouse | SPECTER2 powers graph build; retrieval embeddings are owned by OpenSearch. Reproducibility comes from release-scoped archive artifacts, not a hot PG table. Provenance on graph embeddings stays explicit via `embedding_source_kind`. |
| `paper_evidence_units` lives on warehouse, exposed to serve via FDW | Derives from and joins canonical grounding directly; serve consumes through the bounded FDW path in `00-topology.md` §4. |
| Generated columns for `normalized_title_key`, `normalized_preferred_name`, `normalized_alias`, `fts_vector` | Always consistent with source; no trigger required. |
| `TIMESTAMPTZ` everywhere, `text` everywhere, `jsonb` everywhere | Modern PG defaults. |
| SMALLINT-code enums sourced from `db/schema/enum-codes.yaml` machine-readable registry | `engine/app/models/shared/enums.py` and SQL column comments both generate from the registry; the registry is authoritative, code and comments are derived. |
| `COMMENT ON` every table and non-obvious column | Tooling legibility. |
| No `postgres_fdw` installed on warehouse | Warehouse is source, never querier. |

### Provisional (revisit after schema draft + sample build)

| Decision | Revisit trigger |
|---|---|
| 32-way hash partitioning | Measure partition-lock contention under real 32-way bulk COPY. If clean, keep; if 16 is plenty, shrink. |
| `halfvec(768)` | Confirm recall parity against `vector(768)` on a sample HNSW build once the serving plane is wired. |
| lz4 TOAST on every text column | Measure against raw / pglz on the widest columns (`paper_citation_contexts.context_text`, `paper_blocks.text`). |
| Fillfactor tiers (100 / 90 / 80) | Revisit after first projection-upsert benchmarks. |
| Partial indexes on lifecycle flags | Measure planner uptake; drop any that don't get chosen. |
| BRIN on `ingest_runs.started_at` | Effective only if physical order stays correlated after restores; verify post-restore. |
| `paper_evidence_units` unpartitioned | Partition by `hashtext(evidence_key::text)` × 32 if row count exceeds ~50 M. |
| `paper_chunks` GIN fallback | Remove entirely if the PG fallback retrieval path is retired. |
| `papers (venue_id, year)` composite index | Drop unless the query family proves to use it. |
| `MAXALIGN` ordering confirmed via `pg_column_size` sampling | Already applied in the schema files — verify no accidental misordering after code-gen. |
| `CREATE STATISTICS` on correlated filters (e.g. `paper_citations (intent_code, is_influential, source_mask)`) | Add if planner estimates are off. |

### Deferred (trigger-gated)

| Decision | Trigger |
|---|---|
| `pg_partman` 5.x for `ingest_runs` / `load_history` time-range lifecycle | Row count > 10 M. |
| HNSW on `paper_embeddings_graph` inside PG | A PG-local graph-build query family proves to need it. |
| Columnar extension (Citus / Hydra) on `paper_citation_contexts` | Table > 200 GB and scan-heavy. |
| Citation edge table partitioning switched from `citing_corpus_id` to a composite hash | Reverse-citation workload outgrows the reverse btree. |
| `concept_aliases` trigram index | Fuzzy alias lookup becomes a proven hot path. |
| Staged list-partitioning on `pubtator.*` before `corpus_id` backfill | If `pmid → corpus_id` backfill can't complete in a single pass. |
| `paper_evidence_units` partitioning | Row count > ~50 M. |
| Moving `umls.*` to a read-only replica of a shared UMLS volume | If UMLS reload pressure becomes a contention issue with other warehouse writes. |
